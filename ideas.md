# Ideas & улучшения — Diwo TG Watchdog

Анализ кода по состоянию на март 2026.
Разделено по приоритету: 🔴 критично → 🟡 важно → 🟢 желательно.

---

## 🔴 Безопасность 

### 1. `.env` с реальными секретами не защищён от попадания в git
В проекте нет `.gitignore` и `.env.example`. Реальный токен бота и Fernet-ключ лежат в `.env`. - реализуй

**Что сделать:**
- Создать `.gitignore` (добавить `.env`, `watchdog.db`, `__pycache__/`, `*.pyc`)
- Создать `.env.example` с заглушками вместо реальных значений

## 🟡 Надёжность - не реализуй

### 2. Состояние алертов сбрасывается при перезапуске
`_alert_active` в `widgets.py` — обычный Python-словарь в памяти. После перезапуска бота все алерты считаются новыми и могут прийти повторно. - не реализуй

**Варианты:**
- Хранить состояние `alert_active` в таблице `user_widgets` (добавить колонку `alert_fired_at`)
- Или хранить в отдельной таблице `widget_state(source_id, widget_type, fired_at)`

---

### 3. Исключение в планировщике роняет всю задачу
В `run_scheduler()` (`main.py`) нет `try/except` вокруг `_send_daily_summary()`. Любое необработанное исключение (сеть, БД) завершит задачу планировщика навсегда. - реализуй

**Что сделать:** Обернуть тело цикла в `try/except Exception` с логированием.

---

### 4. Нет ограничения на количество retry в поллере
`_fetch_source_with_retry()` делает повторы при 401, 5xx и сетевых ошибках, но максимальное число попыток нигде не ограничено. При постоянно недоступном API поллер может зависнуть.

**Что сделать:** Добавить `max_retries=3` с экспоненциальным backoff, после исчерпания — логировать ошибку и пропускать итерацию. - реализуй

---

### 5. Нет индексов в БД
Самые горячие запросы (`get_snapshot_around`, `get_widget_subscribers`) делают полный скан таблиц.  - реализуй

**Что добавить в `init_db()`:**
```sql
CREATE INDEX IF NOT EXISTS idx_snapshots_source_ts ON radar_snapshots(source_id, ts);
CREATE INDEX IF NOT EXISTS idx_user_widgets_user ON user_widgets(user_id);
```

## 🟡 Качество кода

### 6. FSM-состояния теряются при перезапуске
Aiogram использует `MemoryStorage` — все незавершённые диалоги (например, пользователь на шаге 4 из 7) сбрасываются при перезапуске бота. -не реализуй

**Что сделать:** Переключить на `RedisStorage` (если есть Redis) или `SqliteStorage` от aiogram-contrib.

---

### 7. URL-поля не валидируются
При добавлении источника бот принимает любую строку в поле API/Auth URL. Ошибка обнаруживается только на этапе тест-запроса, что даёт неинформативное сообщение. - реализуй

**Что добавить:** Простую проверку `url.startswith(("http://", "https://"))` с сообщением "Введите корректный URL".

---

### 8. `get_user_widgets` делает JOIN с `sources`, теряя `daily_summary`
Виджет `daily_summary` хранится с `source_id = 0`, но запрос делает `JOIN sources s ON s.id = uw.source_id` — поэтому `daily_summary` никогда не вернётся из этой функции. Это задокументированное поведение, но может запутать при отладке. - реализуй

**Что сделать:** Добавить комментарий в `get_user_widgets` о намеренном исключении `source_id=0`.

---

## 🟢 Функциональность

### 9. Нет возможности редактировать источник после добавления
После добавления источник нельзя переименовать или изменить интервал опроса. Единственный способ — удалить и добавить заново. - реализуй

**Что добавить:** В детальном виде источника — кнопку "✏️ Изменить интервал" (и/или название).

---

### 10. Нет ручной проверки "прямо сейчас"
Пользователь не может запросить актуальную сводку вне расписания.

**Что добавить:** Кнопку "📊 Сводка сейчас" в детальном виде источника — вызывает `test_fetch_source` и показывает текущий счётчик. - реализуй

---

### 11. Интервал опроса не отображается в UI
При добавлении источника пользователь задаёт интервал, но нигде потом его не видит.

**Что добавить:** Показывать интервал в детальном виде источника (например, `⏱ Опрос: каждые 5 мин`). - реализуй

---

### 12. Новый виджет — «Все офлайн»
Простой и востребованный алерт: все радары источника ушли в офлайн (`online == 0 && total > 0`). - не реализуй

## 🟢 Инфраструктура

### 13. Нет Dockerfile
Развёртывание описано через systemd вручную. Docker упростил бы воспроизводимость окружения и обновления. - не реализуй

**Минимальный вариант:** `python:3.11-slim`, монтировать `watchdog.db` как volume.

---

### 14. Версии зависимостей не зафиксированы полностью
В `requirements.txt` зафиксирован только `aiogram==3.14.0`, остальные (`aiosqlite`, `httpx`, `cryptography`) — без версий. Обновление может сломать совместимость. - реализуй

**Что сделать:** Зафиксировать все версии или добавить `pip freeze > requirements.txt` в инструкцию деплоя.
